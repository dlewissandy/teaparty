### 4. Intent Gap: Failure Recovery Undefined
**Category:** Intent Gap  
**Signal:** When dispatch blocked at proxy gate (needs_plan_review), execution escalated to project-lead, received approval, and re-dispatched with `--auto-approve-plan`. This was reasonable, but when the second dispatch also failed to produce output, no next step was defined. Execution attempted bash searches for files but had no clear recovery procedure.  
**Implication:** Intent must include a **decision tree for failure modes**: 
- If dispatch blocks on planning gate → escalate + re-approve (this was done correctly)
- If dispatch runs but produces no output → [what next?]
- If timeout is exceeded → [what next?]

Current intent only covers the happy path.

## Recommendations for Next Intent Session

1. **Specify execution mechanism explicitly** — dispatch vs. direct — with rationale
2. **Define fallback execution** — if primary mechanism fails, what is the backup?
3. **Create completion gates** — explicit human/system signals between research → writing phases
4. **Specify artifact locations and quality criteria** — where do research outputs live? what does "done" look like?
5. **Add failure recovery decision tree** — for dispatch blocks, missing output, timeouts
6. **Consider serial fallback** — if dispatch overhead is high, can main agent execute research directly with faster handoff?

The intellectual intent (research + write a publication-ready paper using org psych framing) is sound. The operational intent (how to execute multi-phase delegated work) needs more specificity to survive failure.
