
## ADDITIONAL REQUIREMENT: domain field (design gap from original task)

Add a `domain` field to the entry schema everywhere. This field distinguishes task knowledge
from team knowledge and must thread through the full pipeline.

### domain field values: "task" or "team"

Classification rules to add to ALL learning-scope prompts in summarize_session.py:
- domain: team — content is about agent coordination, tool assignment, team workflow,
  dispatch patterns, how agents work together, or agent behavior rules
- domain: task — task-specific facts, domain knowledge, project details, API specifics,
  anything not about team coordination

For `observations` and `escalation` scopes: always set `domain: team` (hardcoded, not haiku-inferred).
Human preference signals and autonomy calibrations are team-composition knowledge.

### compact_memory.py phase retirement must be domain-aware

When doing cross-phase retirement (--current-phase), ONLY retire entries where domain=task.
Entries with domain=team MUST survive phase transitions regardless of their phase field.

- current_phase=implementation: retire phase=specification entries ONLY if domain=task
- current_phase=testing: retire phase=specification OR phase=planning ONLY if domain=task
- domain=team entries: never retired by phase transition (only by prominence < threshold)

### Legacy entries: default domain=task

When compact_memory.py encounters an entry with no frontmatter (legacy format),
assign default domain=task. This is safe — future team entries will have explicit domain=team.

### Additional assertion for test_phase_transition_retires_spec

In test_phase_transition_retires_spec, ALSO create a domain=team entry with phase=specification.
After running compact with --current-phase=implementation, verify the domain=team spec entry
SURVIVES (it must NOT be retired). This validates the task/team distinction.
