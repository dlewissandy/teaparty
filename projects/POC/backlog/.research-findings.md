## R5 — Depth enforcement in cfa_state.py

**Evidence:**
- `/Users/darrell/git/teaparty/projects/POC/scripts/cfa_state.py` lines 77–88 (`CfaState` dataclass with `depth: int = 0` field), lines 108–130 (`make_child_state()` — increments depth with `depth=parent.depth + 1`, no cap check anywhere in the function)
- `/Users/darrell/git/teaparty/projects/POC/docs/POC.md` line 495: "Two levels only (uber + subteams). No recursive hierarchies."

**Severity:** Structural — the two-level constraint is documented as an architectural invariant in POC.md but is not enforced by the state machine. A caller could create depth=2 or deeper child states without any code-level rejection.

## R9 — set_state_direct() used from shell?

**Answer:** `set_state_direct()` is defined in cfa_state.py (lines 293–325) with the docstring: "Directly set a CfaState to a target state, bypassing transition validation. This is the pragmatic escape hatch for the shell orchestration layer..." Its CLI entry point is `--set-state` (line 622: `'--set-state': _cli_set_state`).

Shell scripts **use this bypass extensively and routinely**:
- `intent.sh` defines `intent_cfa_set()` (lines 110–116) which calls `python3 cfa_state.py --set-state`
- `ui.sh` defines `cfa_set()` (lines 400–405) which calls `python3 cfa_state.py --set-state`

`intent.sh` calls `intent_cfa_set()` (bypass) at approximately 9 call sites (lines 229, 326, 343, 364, 380, 386, 394, 452, 458). `plan-execute.sh` calls `cfa_set()` (bypass) throughout the planning and execution phases (PLAN, TASK, WITHDRAWN, WORK_ASSERT, etc.).

The validated `--transition` path is also available via `intent_cfa_transition()` / `cfa_transition()`, but `--set-state` is the **dominant pattern** for state advancement. Most CfA state changes in the shell log as synthetic `'action': 'set-state'` history entries rather than validated transitions.

**Evidence:**
- `/Users/darrell/git/teaparty/projects/POC/scripts/cfa_state.py` lines 293–325 (`set_state_direct`), 558–572 (`_cli_set_state`), 622 (registered as `--set-state`)
- Worktree `intent.sh` lines 110–116 (`intent_cfa_set` wrapping `--set-state`), call sites at approximately lines 229, 326, 343, 364, 380, 386, 394, 452, 458
- Worktree `ui.sh` lines 400–405 (`cfa_set` wrapping `--set-state`)

**Severity:** Structural — the shell orchestration relies on `--set-state` (bypass) far more than `--transition` (validated). The state machine's validation layer is largely circumvented in practice; the transition table's invariants are enforced by code structure and timing rather than by the state machine's own validation logic.

In `docs/GLOSSARY.md` (lines 29–34), "Approval Gate" is described as "a learned confidence model (`scripts/approval_gate.py`) that predicts whether a human would approve a given plan or result. Used to auto-route decisions when confidence is high, and escalate to the human when confidence is low." The GLOSSARY frames it as a routing/prediction component, not as a first-class actor with direct decision authority.

**Evidence:**
- `/Users/darrell/git/teaparty/projects/POC/cfa-state-machine.json` lines 118–124 (WORK_ASSERT transitions, all with `"actor": "approval_gate"`)
- `/Users/darrell/git/teaparty/projects/POC/docs/intent-engineering-detailed-design.md` — no matches for WORK_ASSERT or approval_gate throughout the document
- `/Users/darrell/git/teaparty/projects/POC/docs/GLOSSARY.md` lines 29–34 (Approval Gate described as confidence model/routing aid, not as a CfA actor)

**Severity:** Structural — the detailed design spec is entirely silent on WORK_ASSERT and the approval_gate actor. The JSON designates `approval_gate` as the sole actor with full decision authority (approve, correct, backtrack, withdraw) at the final work review gate, but the spec documentation describes approval_gate as a routing tool that escalates to the human when confidence is low. These two framings are not reconciled: the JSON treats approval_gate as a first-class actor; the GLOSSARY treats it as a confidence-based dispatcher. The discrepancy affects how WORK_ASSERT should behave when the proxy decides to escalate — the JSON provides no `human` actor path out of WORK_ASSERT.
