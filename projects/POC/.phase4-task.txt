Implement Phase 4 of the Hierarchical Agent Memory System: phase transition detection.

## Prerequisites

Phases 1, 2, and 3 must be complete. The system now has:
- YAML frontmatter with phase fields on every entry
- Compaction that retires cross-phase entries when --current-phase is supplied
- Prominence-based retrieval with no evergreen exemption

## What to build

### 1. Create poc/projects/POC/scripts/detect_phase.py

This script detects the current task phase based on task description and optional context.

Purpose: At session start, before memory retrieval, detect whether the session is in
specification, planning, implementation, or testing phase. This phase is used to:
  (a) export POC_TASK_PHASE so compact_memory.py can retire cross-phase entries
  (b) potentially bias retrieval toward phase-relevant memories in future enhancements

Usage:
  python3 detect_phase.py --task "implement the authentication module" [--context-file FILE]

Output: single word on stdout: specification, planning, implementation, or testing
On error or ambiguity: output "operational" (safe default — no cross-phase retirement)

Implementation approach:
1. Use a set of keyword heuristics as a fast first pass (no LLM needed for clear cases):
   - implementation signals: "implement", "write code", "build", "code", "add feature",
     "create file", "fix bug", "patch", "refactor", "develop", "program"
   - testing signals: "test", "verify", "check", "debug", "validate", "run tests",
     "spec compliance", "confirm working"
   - specification signals: "specify", "define requirements", "write spec", "document requirements",
     "design document", "architecture spec", "rfc", "proposal"
   - planning signals: "plan", "design", "architect", "outline", "strategy", "roadmap"
2. If keywords are unambiguous (one category clearly wins), return that phase.
3. If ambiguous (multiple categories match or no keywords match), call claude-haiku with a
   targeted prompt to classify. Haiku receives the task description and returns one word.
4. If haiku call fails, return "operational" (safe default).

The haiku prompt:
  "Classify this task into exactly one phase. Reply with a single word only.
   Phases: specification (writing requirements/design docs), planning (architecture/strategy),
   implementation (writing/editing code), testing (running/writing tests or debugging).
   If none clearly apply, reply: operational

   Task: {task}"

### 2. Edit poc/projects/POC/run.sh to integrate detect_phase.py

READ the file first to get exact line numbers.

CHANGE A: Add phase detection after the memory retrieval block and before the intent gathering
block. Find the section (around line 208) that starts:
  # Memory retrieval (before intent for warm-start)
  MEMORY_CTX_FILE=$(mktemp ...)

After the memory retrieval block completes (after the if block that sets MEMORY_CTX), add:

  # Phase detection
  POC_TASK_PHASE=$(python3 "$SCRIPT_DIR/scripts/detect_phase.py" \
      --task "$TASK" 2>/dev/null) || POC_TASK_PHASE="operational"
  [[ "$POC_TASK_PHASE" =~ ^(specification|planning|implementation|testing|operational)$ ]] \
      || POC_TASK_PHASE="operational"
  export POC_TASK_PHASE
  echo -e "  ${C_DIM}Phase: $POC_TASK_PHASE${C_RESET}" >&2

CHANGE B: The POC_TASK_PHASE must also be added to the environment exported via the SETTINGS_FILE
python block (around line 156-197). Find the 'env' dict in the settings JSON and add:
  'POC_TASK_PHASE': os.environ.get('POC_TASK_PHASE', 'operational'),

This ensures the phase is available to sub-agents (including compact_memory.py calls from
promote_learnings.sh which runs in the parent shell and already has POC_TASK_PHASE exported).

### 3. Write a test: poc/projects/POC/scripts/tests/test_phase4_detection.py

Tests must run with: python3 -m pytest poc/projects/POC/scripts/tests/test_phase4_detection.py -v

Use subprocess to test the CLI, or import detect_phase directly (whichever the implementation supports).
Mock the claude subprocess call to avoid LLM dependency in tests.

test_implementation_keywords:

  Test that "build the new feature" → implementation

test_testing_keywords:

  Test that "debug the authentication bug" → testing
  Test that "verify the compaction logic works" → testing

test_specification_keywords:
