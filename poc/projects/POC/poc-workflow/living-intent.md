# Living Intent: Why One-Time Capture Fails

## The Intent Illusion: Why One-Time Capture Fails

Users often do not know exactly what they want until they see partial output. This is not a failure of communication — it is the nature of complex work. A request is a projection of a need onto language, and that projection is necessarily incomplete. The user who asks for "a technical explainer on our authentication flow" has a real need — they want developers to understand the system — but the request itself does not specify which developers, at what level, with what assumed background, or to what end. Those details exist in the user's head as felt sense, not as articulable requirements. They only become visible when the user sees something that is wrong in a specific and instructive way.

Intent articulation is also lossy in a more structural sense. Tacit knowledge, assumed context, and unstated preferences do not survive the translation into a request. The writer asking for "something professional" carries years of calibrated taste that the word "professional" cannot convey. The product manager asking for "a brief summary" has an implicit length, audience, and register in mind that the word "brief" does not encode. This is what intent engineering addresses when it defines its task as telling agents "what to optimize for, what to protect, and what tradeoffs are acceptable" — but even the best intent engineering session operates on what the user can articulate, not on what they know. The intent gap is not only a session-start problem. It reappears at every major decision point throughout execution, wherever tacit knowledge bears on a choice that the original capture did not anticipate.

The 15-minute intent capture session is necessary but not sufficient. It produces the best hypothesis available given zero execution data, and that is genuinely valuable — it establishes the objective, the success criteria, the decision boundaries, the constraints, and the open questions. But it is still a hypothesis. Execution produces data that the capture session cannot produce: partial outputs, user reactions, discovered constraints, emerging preferences. A system that treats the initial INTENT.md as a finished specification rather than a working hypothesis will optimize for the letter of what was captured while drifting steadily away from the spirit of what was needed.

## Intent as Working Hypothesis

Initial capture is the system's best guess given available information. The INTENT.md it produces — objective, success criteria, decision boundaries, constraints, open questions — is the most informed hypothesis possible before any work has been done. The appropriate relationship to that document is not reverence but scientific: it encodes a testable model of what the user wants, and execution will begin generating evidence about whether that model is accurate.

Planning refines the hypothesis before execution generates evidence, because decomposition forces specificity. When you break "write a technical explainer" into sections, you immediately discover latent questions: for what audience? at what assumed depth? with what prerequisites? These questions were present in the original request — they were simply invisible at the level of abstraction the request occupied. The planning phase surfaces them, and each one is an opportunity to either resolve an open question or to explicitly flag an assumption that will need to be tested against user feedback.

Execution confirms or challenges the hypothesis at every step. Each partial output is an intent probe. The user who says "this is good but a bit formal" has just revealed that the original capture's assumption about register was wrong — and that revelation carries more information than any amount of additional questioning at the start of the session could have produced. The user who says "this is exactly right" has confirmed an assumption that the system should weight more heavily going forward. In both cases, the partial output did something the initial capture could not: it gave the user something specific to react to, and reactions are more precise than requests.

Delivery validates whether the output satisfied the purpose or merely the stated request. These diverge regularly, and the divergence is often only visible in retrospect. A blog post that checks every box in the original INTENT.md but fails to drive the intended behavior — developer adoption, stakeholder buy-in, team alignment — has satisfied the letter of the intent while missing its point. The governing metric is not completion of the stated deliverable but "the system's ability to produce work that reflects what the human actually wanted improves with each interaction." Delivery is where that metric is ultimately tested.

## The Intent Delta Check

At each major execution milestone, the system should re-read INTENT.md's success criteria and ask a single operational question: is the current trajectory still serving this? Not "are we producing good work" and not "are we following the plan" — but specifically, is the work we are doing still aimed at the purpose that was captured at the start? This check is not overhead. It is the mechanism that prevents the system from optimizing a path that has already diverged from the destination.

The divergence types that this check catches fall into three categories:

1. Scope drift: doing more or less than intended
2. Approach drift: solving the right problem the wrong way
3. Purpose drift: the stated goal is no longer aligned with the user's deeper need

The decision criteria for each type are not categorical but continuous, and they follow a single governing formula: magnitude of divergence multiplied by reversibility of the current path. A small divergence on a reversible path warrants continuing and noting — the cost of pausing to confirm exceeds the cost of a minor correction at delivery. A large divergence or an irreversible path warrants pausing — the cost of discovering the problem late is disproportionate to the cost of a mid-execution check. This asymmetry is the same one that governs the escalation calibration model, which weights negative signals at −0.10 against positive signals at +0.05. The logic is identical: the cost of a false negative (missing a real problem) exceeds the cost of a false positive (pausing unnecessarily). Applied to intent drift, this means that any divergence that is large or irreversible should trigger a pause and confirm, while small and reversible divergence earns a note and a continuation.

Purpose drift is the most dangerous type because it is the hardest to detect and the most expensive to correct late. When the stated goal has shifted — when the blog post is no longer about developer adoption but about something else the user has not said yet — no amount of execution quality recovers the purpose. Purpose drift requires a revision to INTENT.md itself, not just a course correction within the current plan.

## Stated vs. Revealed Preferences

What users say they want and what their feedback patterns reveal they actually value are consistently different things, and the delta between them is the most important learning signal in the system. The spec's recommendation to surface contradictions explicitly rather than silently auto-correct is correct — silent correction trains the user to assume alignment they have not verified, and it deprives the system of the confirmation signal that a stated preference actually matches a revealed one. But the recommendation is incomplete. Surfacing the contradiction in the moment is necessary; tracking the delta over time is what makes the system learn.

The escalation calibration model tracks confidence adjustments: +0.05 for a positive signal, −0.10 for a negative signal, −0.02 for a neutral one. This asymmetric weighting should be extended to intent signals. When a user confirms that an approach was right, that confirmation should carry weight in the intent memory — it narrows the hypothesis space for future sessions. When a user corrects an approach, that correction should widen the hypothesis space and update the prior. A confirmed approach reduces uncertainty; a corrected approach increases it and redirects it. Neither signal is wasted, but neither is the correction more important than the confirmation. Both are intent data. Only one of them is currently being tracked consistently.

Divergence between stated and revealed preferences is the system's long-term honesty test. An agent that consistently produces work requiring correction is not learning — it is re-running prior assumptions against contradicting evidence and ignoring the evidence. The governing metric demands improvement over time: not just accurate capture at session start, but narrowing divergence across sessions. That improvement only accrues if the system treats revealed preferences as first-class data, not as edge cases to handle gracefully and then forget.

## Living Intent in Practice

The following example traces a single content task through four intent revisions. Each revision is not a failure — it is the process of intent becoming legible through execution.

1. Initial request: "Write a blog post about our new API." INTENT.md captures: objective is developer awareness, audience is assumed general-technical, tone is assumed brand-consistent. Success criteria: a published post that drives API adoption. The hypothesis is thin because there is no execution data yet.

2. After the first draft, the user says: "This is good but it reads too casual — our developers expect more technical depth." INTENT.md revision: audience specification tightens from general-technical to practicing developers; a formal-technical tone constraint is added. Divergence type: approach drift — right problem, wrong register. Signal to the memory system: this user's developer content skews more formal than default assumptions; update the prior.

3. After the revised draft, the user says: "Actually, can you add code examples? Developers will want to see how it works." INTENT.md revision: success criteria now explicitly includes functional code examples in at least two languages; the decision boundary is updated to require code validation before delivery. Divergence type: scope drift — the original request underspecified the deliverable. Signal to the memory system: for this user's developer-facing content, code examples are a standing expectation, not an optional enhancement.

4. After the final review, the user says: "We need this to match our house style guide." INTENT.md revision: the constraints section gains a style guide reference as a hard constraint, not a preference. Divergence type: approach drift again — the approach was producing good content while bypassing an existing organizational standard. Signal to the memory system: this user operates within institutional constraints that should be surfaced at the start of any content task, not discovered at delivery.

None of these corrections were failures. They were the natural process of intent becoming legible through execution. A system that treated the first capture as final would have delivered a technically correct but purposively wrong output. The warm start protocol's framing applies precisely here: corrections to pre-populated elements are high-value signal — the model has diverged from reality, and the correction is how reality reasserts itself. Living intent is not a design concession to user imprecision. It is the mechanism by which the system stays honest.
