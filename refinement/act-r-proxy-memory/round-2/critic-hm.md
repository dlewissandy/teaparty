# Hiring Manager Review — Round 2

## Concerns

### 1. The evaluation metrics have no success thresholds or sample size requirements
[evaluation] The shadow mode now lists four metrics (action match rate, prior calibration, surprise calibration, retrieval relevance) and four ablations. Good. But none of them have go/no-go criteria. What action match rate justifies moving to Phase 2? 70%? 90%? If multi-dimensional embeddings beat single-embedding by 2% on retrieval relevance, is that enough to justify the 5x embedding cost? The ablation plan says what to compare but not how to decide. With 50-200 lifetime interactions (the document's own estimate), statistical power is a real problem — you may not have enough data to distinguish the ablations from noise. The evaluation section needs: (a) minimum sample sizes per ablation, (b) effect size thresholds for each comparison, and (c) a decision rule for what happens if the metrics are ambiguous. Without these, shadow mode produces data that nobody knows how to interpret. (act-r-proxy-memory.md, Migration Path)

### 2. The retrieval score conflates threshold and ranking into one number
[feasibility] The combined score `activation_weight * normalize(B) + semantic_weight * cosine(...) + noise` is used for two distinct purposes: thresholding (discard if score < tau) and ranking (return top-k). But tau = -0.5 was chosen as an ACT-R activation threshold on the unnormalized B scale. After normalization of B to [0, 1] and addition of a cosine component bounded [-1, 1], the combined score occupies a completely different range than raw activation. Applying the same tau = -0.5 to the combined score has no theoretical grounding — it is a number chosen for one scale applied to another. The retrieve() function in act-r-proxy-mapping.md applies tau to the combined score (line 363: `if score > tau`), which means the threshold is now filtering on a mixture of normalized activation, similarity, and noise, not on activation alone. This needs to be rethought: either threshold on raw B before scoring (filter by memory accessibility, then rank by relevance) or pick a new threshold calibrated to the combined score's actual range. (act-r-proxy-mapping.md, retrieve function and combined retrieval score)

### 3. The multi-dimensional similarity average rewards sparse chunks unfairly
[feasibility] The retrieval_score function averages cosine similarities across matched dimensions only — if a chunk has embeddings for situation and stimulus but None for artifact, response, and salience (because it was a non-surprise approval with minimal content), the average is over 2 dimensions. A richly populated surprise chunk averages over 5. The average operation means the sparse chunk's score is determined by fewer, potentially noisier signals, while the rich chunk's score is regularized by more dimensions. But there is no penalty for sparsity — a chunk matching well on 2 of 2 populated dimensions scores the same as one matching well on 5 of 5. The document says "chunks matching on more dimensions score higher (intersection effect)" (act-r-proxy-mapping.md, retrieval_score docstring), but that is only true if the additional dimensions contribute positive similarity. If the design intends intersection semantics, the aggregation should account for the number of possible dimensions, not just the number of populated ones — e.g., divide by 5 always, or weight by dimension count. (act-r-proxy-mapping.md, retrieval_score implementation)

### 4. No estimate of how long shadow mode needs to run
[evaluation] The migration path describes three phases but gives no timeline for Phase 1. The system produces 50-200 lifetime interactions. Shadow mode needs enough interactions to (a) populate the memory with chunks across multiple task types and CfA states, (b) observe enough surprises to test the salience mechanism (at the document's own "most gates produce no surprise" admission, this could take many sessions), and (c) run the four ablations with meaningful sample sizes. Is this 2 weeks? 3 months? A year? The answer matters because shadow mode doubles LLM costs (running both old and new systems in parallel) while delivering zero user-facing value. If shadow mode needs 6 months of parallel running to produce enough data, that changes the funding calculus significantly. (act-r-proxy-memory.md, Migration Path)

### 5. The REINFORCE step creates a systematic retrieval bias toward old chunks
[risk] The session lifecycle reinforces every loaded chunk at every gate (the REINFORCE step in the pseudocode). Chunks loaded at session start — the ones with highest activation — get a trace at every gate interaction throughout the session. A session with 5 gates adds 5 traces to each loaded chunk. This creates a rich-get-richer dynamic: the chunks that are already most active get the most reinforcement, making them more likely to be loaded next session, where they get reinforced again. New chunks from recent sessions compete against chunks that have accumulated dozens of traces from the reinforcement loop. The document removed the dangerous "explicit reinforcement" rule 3 from round 1 — but the REINFORCE step in the session lifecycle is a milder version of the same problem. It is standard ACT-R (retrieval creates traces), but in ACT-R a chunk is only reinforced when it is specifically retrieved for a task, not blanket-reinforced on every cycle. Loading 30 chunks into the prompt prefix and reinforcing all 30 at every gate is not selective retrieval — it is broadcast reinforcement. (act-r-proxy-memory.md, Session Lifecycle pseudocode, REINFORCE step)

### 6. Upstream context has no retrieval pathway
[missing] The sensorium document lists "upstream context" as a percept the proxy senses (line 14: "the intent behind the plan, the plan behind the deliverable"). The round 1 quality evaluation flagged this as a minor coherence gap (the "upstream" embedding dimension was removed but upstream context is still listed as a sensory input). Draft-1 did not address it. Where does upstream context go? If it folds into the artifact embedding, say so. If it is passed as raw text in the prompt but not embedded, say so. Currently the proxy senses upstream context, uses it in Pass 2 (the posterior prompt includes "upstream context"), but has no memory pathway for it — meaning the proxy cannot retrieve past interactions by upstream similarity. For a system where the same intent can produce multiple plans across sessions, this seems like a significant missing dimension. (act-r-proxy-sensorium.md, The Problem section; act-r-proxy-mapping.md, chunk schema)

## What Improved Since Round 1

The normalization of the combined retrieval score is the most important fix — the original scale mismatch would have made the embedding system inert in practice. The removal of trace rule 3 (explicit reinforcement by outcome similarity) eliminated a genuine feedback bomb. The graded surprise mechanism (confidence delta > 0.3 as a moderate trigger) captures real information the binary mechanism lost. The evaluation metrics and ablation plan, while still underspecified, transform the migration path from "trust us" to something that can be held accountable. The parameter provenance corrections (s and tau as design choices, d = 0.5 as a starting point) make the document honest about what is established science vs. design judgment. The published precedent citations (Park et al., Nuxoll & West, Bhatia et al.) replace hand-waving with verifiable evidence. Overall, the document moved from "interesting theory paper" to "plausible engineering proposal" in one round.

## Bottom Line

Closer to fundable, but not there yet. The engineering gaps from round 1 are mostly fixed. The new gaps are in evaluation design and in subtle interaction effects that emerge when you think through the system running over many sessions. The REINFORCE broadcast problem (#5) is the most concerning — it is a feedback loop that could make the memory system ossify around early interactions, which directly undermines the "drifting interests" behavior the design claims to produce. The threshold mismatch (#2) is a straightforward bug that needs fixing before implementation. The evaluation plan (#1, #4) needs to be concrete enough that someone other than the designer can run shadow mode and know when it has succeeded or failed. Fix these four things and I would fund a Phase 1 build.
