# Defense -- Round 2

## Concessions

### 1. Fabricated citations are indefensible
The critic is right. The synthesist hallucinated author names for two of three new citations introduced in draft-1. "Nuxoll & West (2024)" are fabricated authors for what is actually Honda, Fujita, Zempo, & Fukushima (2025), HAI '25. "Bhatia et al. (2026)" is a fabricated first author for what is actually Meghdadi, Duff, & Demberg (2026). The factcheck identified both fabrications with correct DOIs and real author names. These citations were introduced specifically to replace the unverifiable "deployed at scale" and "Claude memory" claims from draft-0 -- replacing unverifiable claims with fabricated citations is a regression, not a fix. The irony is that the underlying papers are real and do support the document's argument; only the attribution is wrong. This must be corrected with the factcheck's verified citations. The Meghdadi et al. paper's description should also be revised -- it is a psycholinguistics study about associative priming in the Lexical Decision Task, not about agent memory architecture. The relevance is the LM-embedding-as-ACT-R-association-metric pattern, and the description should say so honestly. Park et al. (2023) is confirmed correct (minor formatting: "Park, J.S." not "Park, S.").

### 2. salient_percepts field is missing from the data model
The critic is right. Both the sensorium document and the mapping document's JSON example show `"salient_percepts": ["no rollback strategy", "database migration risk"]` as a list field in the chunk. But the MemoryChunk dataclass has no `salient_percepts` field. The SQL schema has no `salient_percepts` column. The `record_interaction()` function has no `salient_percepts` parameter. This is a straightforward consistency bug: the field appears in prose examples but was never added to the implementation sketch. The fix is to add `salient_percepts: list[str]` to the dataclass, a `salient_percepts TEXT` column (JSON array) to the SQL schema, and a corresponding parameter to `record_interaction()`. The field is populated only when surprise is detected (strong or moderate), matching the graded surprise mechanism from draft-1.

### 3. Tau semantics mismatch between theory and implementation
Both the HM and the logic critic identified this independently, which is a strong signal it is real. The theory document (act-r.md) defines tau as a threshold on raw activation A. The implementation applies tau to the combined score (normalized activation + cosine similarity + noise). These are different quantities on different scales. A tau of -0.5 applied to raw B means "admit chunks with slightly negative activation." A tau of -0.5 applied to the combined score (where normalized B is in [0,1] and cosine is in [-1,1]) means something entirely different -- and since the combined score with 0.5/0.5 weights and no noise ranges roughly from -0.5 to 1.0, a threshold of -0.5 admits essentially everything.

The logic critic further notes that the soft-threshold retrieval probability equation from act-r.md (`P(retrieve) = 1 / (1 + exp(-(A - tau) / s))`) is presented as the governing retrieval model but is never implemented -- instead a hard cutoff is used on the combined score. These are two different retrieval mechanisms sharing the same symbol.

The fix should be: threshold on raw B first (filter by memory accessibility), then score the survivors with the combined formula, then rank. Tau applies to B; the combined score is for ranking only. The soft-threshold equation can be used as the gating function on raw B, which is where it belongs theoretically. The combined score then needs its own top-k cutoff but not a theoretically-grounded threshold. This separates the two operations that the current design conflates.

### 4. Multi-dimensional cosine average does not produce the claimed intersection effect
The logic critic is right for the second round running. Round 1 removed the "low-fan spreading activation" analogy. Draft-1 replaced it with a claim that averaging cosine similarities across matched dimensions produces an "intersection effect" where "chunks matching on more dimensions score higher." The arithmetic directly contradicts this: a chunk matching well on 2 of 2 populated dimensions (average 0.85) outscores a chunk matching moderately on 5 of 5 (average 0.30). Averaging rewards high similarity on few dimensions, not breadth of matching. The docstring in `retrieval_score()` makes the false claim directly.

The HM's framing of the same issue is also correct: sparse chunks (non-surprise approvals with 2 populated dimensions) get their average computed over fewer, potentially noisier signals, with no penalty for sparsity. If the design intends intersection semantics, the aggregation must change. Options: (a) divide by the total number of possible dimensions (5), not the number of populated ones -- this penalizes sparsity and rewards breadth; (b) use a sum rather than an average, so more matching dimensions produce higher scores; (c) use a weighted combination where the dimension count itself is a factor. Option (a) is simplest and directly produces the claimed behavior: a chunk matching well on 5/5 dimensions scores higher than one matching equally well on 2/5. The docstring and prose must be corrected to match whichever aggregation is chosen.

### 5. The REINFORCE step is broadcast reinforcement, not selective retrieval
Both the HM and the logic critic flagged this independently. The session lifecycle adds a trace to every loaded chunk at every gate. This is not standard ACT-R behavior -- in ACT-R, a chunk is reinforced only when it is specifically retrieved for a task, not when it happens to be sitting in the buffer. Loading 30 chunks into the prompt prefix and reinforcing all 30 at every gate is broadcast reinforcement. It creates a rich-get-richer feedback loop: the most-activated chunks get loaded, get reinforced by every gate regardless of relevance, and remain most-activated next session.

The proponent's round-1 defense successfully argued that trace rule 3 (explicit reinforcement by outcome similarity) was "a bias bomb" and had it removed. The REINFORCE step has an analogous feedback structure -- reinforcement by loading rather than by retrieval relevance. The logic critic correctly distinguishes this from round 1's HM #11: this is a popularity bias, not an outcome bias, but the runaway dynamics are similar.

The fix: reinforce only chunks that the retrieval function actually scored above a relevance threshold for the specific gate, not all loaded chunks. Alternatively, reinforce loaded chunks once at session start (they were relevant enough to load) but not at every gate. The per-gate blanket reinforcement is the problematic pattern.

### 6. Evaluation metrics need go/no-go criteria and sample size estimates
The HM is right that the evaluation plan lists what to measure but not how to decide. With 50-200 lifetime interactions, statistical power is a real constraint. The ablation plan says what to compare but provides no effect size thresholds, minimum sample sizes, or decision rules for ambiguous results. Shadow mode will produce data that nobody knows how to interpret. The evaluation section needs: (a) a minimum interaction count before any ablation comparison is meaningful (likely 50+ per cell, which constrains the timeline); (b) concrete go/no-go thresholds (e.g., action match rate above 70% to proceed to Phase 2); (c) a decision rule for the multi-dimensional embedding ablation that accounts for the 5x embedding cost.

## Defenses

### 1. The cold-start argument is not circular
The logic critic says the cold-start section "uses its own conclusion as evidence" -- that claiming the new system's cold start is "more informative" assumes chunk-based data is more valuable than scalar updates, which is the conclusion the document set is trying to establish.

This misreads the argument. The cold-start claim is narrower than the critic reads it. The claim is not "chunks are better than scalars" (which is indeed the document's broader conclusion). The claim is: "with zero memories, the proxy must ask the human everything, which generates rich interaction data; with a below-threshold EMA, the proxy escalates silently, which generates a single binary outcome." These are mechanically different cold-start behaviors. The first produces dialog (questions, responses, reasoning), which creates chunks with content, embeddings, and structure. The second produces one bit (approve/reject). The observation that the first cold start generates richer training signal is not the document's contested conclusion -- it is a direct consequence of the two systems' mechanical behavior when memory is empty. Even if the ACT-R retrieval system turned out to be worse than EMA once populated, the cold-start phase would still produce more data per interaction.

### 2. Min-max normalization is query-dependent by design
The logic critic identifies that normalization over the candidate set makes a chunk's activation score depend on what other chunks are in the query. This is true and is the correct behavior, not a bug. The purpose of normalization is to make activation and similarity contribute on comparable scales for a specific retrieval decision. The relevant question at retrieval time is not "what is this chunk's absolute activation?" but "how does this chunk's activation compare to the other candidates for this query?" Min-max normalization answers the relative question. Park et al. (2023) use exactly this approach for the same reason -- their recency, importance, and relevance scores are normalized per-query, not globally.

The critic's specific concern -- that queries returning chunks with similar activations make activation nearly uniform and the score becomes dominated by semantic similarity -- is a feature. When all candidate chunks have similar activation (they are equally accessible), semantic relevance should dominate. When activation varies widely (some chunks are much more accessible than others), activation should have strong discriminating power. The query-dependent weighting produces the right behavior in both cases.

That said, the 0.5/0.5 starting point should carry a note that its effective influence varies by query. This is a documentation improvement, not a design change.

### 3. The LLM will condition on retrieved memories
The logic critic flags an unstated assumption: the LLM might override retrieved memories with its pretraining priors. This concern is theoretically valid but practically manageable. LLMs are well-documented to condition strongly on in-context examples -- this is the entire basis of few-shot prompting, retrieval-augmented generation, and the document's own approach of injecting memories into the prompt. The proxy's memories are presented as specific, concrete episodes ("at PLAN_ASSERT for the migration project, the human said: add a rollback strategy"). Specific episodic examples in context consistently override general pretrained dispositions in current LLMs. This is not controversial -- it is why RAG works at all.

The edge case where memories are sparse or contradictory and the LLM falls back to its own judgment is real but is the cold-start problem, which the document addresses: with few memories, the proxy escalates to the human. As memories accumulate, they provide stronger in-context signal. The assumption is not that the LLM is a "transparent reasoner" -- it is that the LLM conditions on specific episodic examples in its context, which is an empirical property that the entire field of prompt engineering depends on.

### 4. Short text fragments produce useful embeddings
The logic critic questions whether embedding "PLAN_ASSERT security" (two tokens) or a single sentence produces vectors with useful cosine-similarity geometry. Modern embedding models (including the ones available through the Anthropic API) are specifically trained and evaluated on short text pairs -- the MTEB benchmark includes sentence-level and even phrase-level similarity tasks. The STS (Semantic Textual Similarity) benchmark, which is central to embedding model evaluation, consists entirely of sentence pairs. Embedding models perform well on short text by design; this is their primary use case, not an edge case.

The situation embedding ("PLAN_ASSERT security" vs. "PLAN_ASSERT documentation") is a valid concern -- these are label-like inputs where the semantic distance may not be well-calibrated. But this dimension is also covered by structural filtering (exact SQL match on state and task_type), so the embedding is a secondary signal. The dimensions where embedding quality matters most (artifact, response, salience) contain natural language sentences or paragraphs, which are squarely in the regime where embedding models excel.

### 5. The confidence threshold (0.3) is a starting heuristic, not a precision instrument
The logic critic correctly notes that LLM-generated confidence scores are poorly calibrated and that a threshold of 0.3 on uncalibrated scores may trigger on noise. This is a valid concern but does not undermine the graded surprise mechanism. The threshold is a design parameter to be calibrated in shadow mode, just like d, s, and tau. The document already says so for those parameters; the confidence threshold should carry the same caveat. The mechanism itself -- capturing cases where the artifact was noteworthy enough to shift confidence substantially without changing the action -- is sound in principle. Whether 0.3 is the right threshold, or whether LLM confidence values are stable enough to threshold on at all, is an empirical question for shadow mode. If confidence values prove too noisy, the fallback is the binary mechanism (action change only), which is more robust precisely because it thresholds on a categorical variable rather than a continuous one.

## Clarifications

### 1. Upstream context has a retrieval pathway -- it folds into existing dimensions
The HM identifies that the sensorium document lists "upstream context" as a percept the proxy senses (line 14) but has no dedicated embedding dimension for it after the "upstream" dimension was removed in draft-1. This is a documentation gap, not a missing capability. Upstream context enters retrieval through two existing pathways: (a) the artifact embedding, since the artifact is evaluated against upstream context and the embedding captures their relationship; and (b) the stimulus embedding, since the gate question often references the upstream source ("does this plan address the intent's success criteria?"). A dedicated upstream dimension was considered in the anchor (the sensorium's original 6-dimension table included it) and was removed during round-1 harmonization because it overlapped with artifact and stimulus without adding discriminating power. The sensorium document should state this explicitly -- that upstream context is sensed but enters memory through the artifact and stimulus embeddings rather than via a dedicated dimension -- rather than leaving the reader to wonder where it went.

### 2. Shadow mode timeline depends on the system's interaction tempo, not calendar time
The HM asks how long shadow mode needs to run and notes that with 50-200 lifetime interactions, the timeline matters for cost planning. This is a fair operational question, but the answer is interaction-count-based, not calendar-based -- which is consistent with the document's own design philosophy. Shadow mode needs enough interactions to populate memory across the major CfA states and task types (rough minimum: 50 interactions spanning at least 3 task types and 4 CfA states). In a system processing 5-10 gates per session with 2-3 sessions per week, this is approximately 3-5 weeks. In a system with lower tempo, it takes longer. The document should specify the minimum interaction count and the coverage requirements rather than a calendar duration. The cost concern (doubling LLM costs during shadow mode) is real but bounded: shadow mode runs only the proxy's calls twice, not the entire orchestrator. The proxy's LLM spend is a fraction of the total session cost.
