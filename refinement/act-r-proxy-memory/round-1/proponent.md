# Defense — Round 1

## Concessions

### Embedding dimensions are inconsistent across documents (Logic #1)
The critic is right. The root document's KV cache MemoryChunk omits `embedding_artifact`; the sensorium adds `upstream` as a sixth dimension that appears nowhere else. The mapping document's 5-dimension schema should be canonical -- it is the one with the implementation sketch and the retrieval function. The root document's KV cache section and the sensorium table need to be reconciled to match. The `upstream` dimension in the sensorium document should either be added to the mapping schema with a retrieval path or removed from the sensorium table.

### Noise parameter s = 0.25 is not the ACT-R standard (Factcheck)
The critic is right. The ACT-R default for `:ans` is NIL (disabled), and the tutorial value when enabled is 0.5, not 0.25. The document should either cite a specific model-fitting source for 0.25 or use 0.5 as the starting value and note that it needs calibration. Given that the noise is additive to a combined score that already departs from standard ACT-R (the linear combination with cosine similarity), the specific value matters less than the document claims -- but the claim of "standard value" is wrong and should be corrected.

### Retrieval threshold tau = -0.5 is not the ACT-R standard (Factcheck)
The critic is right. The ACT-R default for `:rt` is NIL, and the tutorial demonstration value is 0. The document should state that -0.5 is a design choice for this system, not the canonical ACT-R value, and explain why a negative threshold is appropriate (it admits chunks with slightly negative activation, which may be desirable in a low-interaction system where many useful chunks hover near zero).

### Cost model arithmetic errors for G=10 and G=20 (Factcheck)
The critic is right. G=10 should be 62,300 (not 62,600) and G=20 should be 118,300 (not 119,200). The savings percentages are approximately correct despite the wrong totals, but the numbers need fixing. The 376-chunk figure also needs its arithmetic basis stated (it implies ~12K tokens reserved for system prompt, which is plausible but should be explicit).

### Cost model omits output tokens and cache-write premium (Logic non sequitur #2)
The critic is right. The cost model counts input token-equivalents for prefix reuse but omits: output tokens (which double with two passes), the 1.25x cache-write premium on the first call, and the full cost of surprise-extraction calls. The conclusion that two-pass-with-cache is cheaper than single-pass may still hold -- the prefix savings are large -- but the model as written understates the true cost. The cost section should include output tokens and the cache-write premium, then re-derive the comparison. If the conclusion weakens, state that honestly.

### No evaluation criteria or success metrics (HM #1)
The critic is right. The migration path says "log and compare" without defining what comparison means. The document needs: (1) a concrete metric for shadow mode -- at minimum, action match rate (did the proxy's posterior match what the human actually did?) and prior calibration (how often did the prior match the posterior?); (2) an ablation plan that isolates multi-dimensional embeddings vs. single, ACT-R decay vs. simple recency, and two-pass vs. single-pass. Without these, you cannot tell whether the complexity is earning its keep.

### Embedding cost is buried (HM #8)
The critic is right. Up to 5 embedding calls per chunk creation, plus retrieval-time embedding of the current context, is a real cost that the budget section ignores. This should be quantified and included in the cost model. Embedding calls are cheap relative to LLM generation calls, but "cheap" is not "free" and the omission undermines the cost model's credibility.

### No discussion of memory maintenance or garbage collection (HM #10)
The critic is right. The traces list grows without bound, and the REINFORCE step (add a trace to every loaded chunk at every gate) accelerates this. After 1000 sessions with 5 gates each and 30 loaded chunks, a long-lived chunk could have thousands of traces. The design needs a compaction strategy -- e.g., replace traces older than some horizon with a summary count, or cap the traces list and merge old traces. The SQLite storage also needs a pruning policy for chunks that have been below threshold for N sessions. This is an operational requirement, not a nice-to-have.

### "Explicit reinforcement" criterion is unspecified (HM #11)
The critic is right. Trace creation rule 3 ("when a new interaction produces a similar outcome") is dangerously vague. If "similar" means exact match on (state, outcome), then every approval at PLAN_ASSERT reinforces every prior approval at PLAN_ASSERT -- a runaway positive feedback loop. This rule should either be removed entirely (let retrieval-based reinforcement handle it, which is rule 2) or given a precise, narrow criterion (e.g., cosine similarity above a high threshold on the response embedding). The current wording is a bias bomb.

### Chunk creation ambiguity for non-surprise gates (Logic #4)
The critic is right. The mapping document says every interaction produces a chunk. The root document's session lifecycle is ambiguous about whether non-surprise gates produce chunks. These must agree. The answer should be: every gate produces a chunk (with structural fields, prediction fields, and outcome), but non-surprise chunks have empty salience fields. The lifecycle pseudocode should make this explicit in the STORE step.

## Defenses

### The combined score mixes incompatible scales (HM #3)
This is a real engineering problem, but the document's design is not broken -- it is a starting point that needs calibration, and the document says so. The 0.5/0.5 weighting is explicitly labeled "starting point: calibrate empirically." That said, the critic is right that the scale mismatch means the starting point is not actually 50/50 influence. The fix is straightforward: normalize B to [0, 1] by applying a sigmoid or by dividing by an empirically-determined range (e.g., B values in practice range from roughly -2 to +3 for this system's interaction volumes). This is a calibration task, not a design flaw. The architecture of "weighted combination of activation and similarity" is sound; the specific normalization needs to be specified. Concretely: add a normalization step to the retrieval_score function, and commit to defining the normalization range during Phase 1 shadow mode when real activation distributions are observed.

### d = 0.5 may not transfer to low-volume interaction streams (HM #4, Logic non sequitur #3)
The hiring manager and logic critic both flag this. They are right that the regime is different -- proxy interactions are sparse compared to newspaper headlines. But the document's argument for d = 0.5 is not "the proxy's interactions have the same statistical structure as headlines." It is: (a) the power-law form is well-established as the right functional shape for memory decay, and (b) 0.5 is a reasonable starting exponent because it produces moderate decay -- not too aggressive (d = 1.0 would make even recent memories fade fast) and not too conservative (d = 0.1 would make everything persist). The document should acknowledge that d = 0.5 is a starting value informed by ACT-R's empirical tradition, not a value validated for this specific domain. The interaction-based time substitution is defensible on its own merits (no decay during idle time is correct for an agent), even if the specific exponent needs tuning. The Anderson & Schooler citation supports the power-law form and the event-based framing; whether 0.5 is optimal for proxy-scale interaction volumes is an empirical question the document should flag rather than treat as settled.

### The two-pass prediction assumes LLM consistency (HM #6)
The concern is valid but overstated. Temperature 0 does not guarantee determinism, but it does substantially reduce variation. The relevant question is not whether the delta is noise-free, but whether the signal-to-noise ratio is high enough to be useful. The two passes differ not just in "phrasing differences" but in a substantive content difference: Pass 2 includes the entire artifact text. An LLM that cannot produce meaningfully different predictions when given vs. not given a multi-page plan document would be useless for any conditional generation task. The design should add a simple validation: if the prior and posterior actions match, the confidence values are close, and the prose reasoning is semantically similar, treat this as "no surprise" regardless of minor textual differences. But the core assumption -- that providing vs. withholding the artifact produces meaningfully different LLM outputs -- is well-supported by how LLMs actually work. The concern about prompt-sensitivity noise is a reason to filter the delta signal, not a reason to abandon two-pass prediction.

### Surprise is binary but attention is not (HM #7)
The critic identifies real information loss, but the design choice is deliberate and defensible. The binary trigger (action changed vs. didn't) is a simplification that trades completeness for signal quality. When the action changes, the delta is unambiguous -- something in the artifact flipped the decision. When the action doesn't change but confidence shifts, the signal is noisier -- was the confidence shift meaningful or just LLM output variation? Extracting salient percepts from a confidence-only shift requires interpreting why the LLM became more or less confident, which is less reliable than interpreting why it changed its recommended action. That said, the critic is right that a case where confidence jumps from 0.3 to 0.9 without changing the action is informative. A reasonable middle ground: trigger surprise extraction when the action changes OR when the confidence delta exceeds a threshold (e.g., |posterior_confidence - prior_confidence| > 0.3). This captures the high-signal cases without opening the door to noise from minor confidence fluctuations.

### The KV cache section is speculative (HM #9)
The critic says ~40% of the memory document is devoted to a future design that can't be built yet. This is true, and the section is clearly labeled as future work. But the section serves a purpose beyond speculation: it demonstrates that the current text-retrieval design and the future KV-cache design share the same memory theory and differ only in delivery mechanism. This matters because it shows the current design is not a dead end -- it is the first phase of an architecture that scales. The cost model in the KV section should be clearly separated from the core design's economics. The core design (text retrieval, no caching) should have its own cost analysis that stands independently. The KV section's cost model should be presented as "here is what becomes possible when we migrate to direct API calls," not as part of the justification for the current design.

### Whether the proxy can auto-approve (Logic #3)
The critic identifies a real tension but misreads it as a contradiction. The root document criticizes auto-approval via EMA because EMA skips the dialog -- it never inspects the artifact, never asks questions, just checks a scalar and waves things through. The sensorium document's auto-approval is fundamentally different: the proxy has run both passes, inspected the artifact, identified that nothing surprising is present, and confirmed that its model of the human's attention patterns predicts approval. The dialog happened -- it happened inside the proxy's two-pass reasoning. The root document's critique is about skipping inspection. The sensorium's auto-approval is about completing inspection and finding nothing to escalate. These are not the same thing. However, the documents should make this distinction explicit rather than leaving readers to reconcile them. The root document should acknowledge that auto-approval is acceptable when the proxy has demonstrably inspected the artifact and its prior-posterior agreement reflects genuine understanding, not pattern-matching on a scalar.

### Cross-task learning may contaminate rather than help (Logic unstated assumption #3)
The critic assumes that learning from dispatch A (security) has no relevance to dispatch B (documentation). This is often true but not always. Cross-cutting patterns exist: "this human always asks about test coverage" applies regardless of task type. The serial processing ensures that if the security review reveals a human preference that generalizes (e.g., "always wants rollback plans for anything touching production"), it is available for subsequent gates. Whether this produces useful priming or interference is indeed an empirical question, but the structural filtering in retrieval already handles the concern: when processing dispatch B (documentation), the retrieval query filters on task_type, so security-specific chunks are not surfaced unless the semantic match is strong. The FIFO queue is about ordering, not about forcing cross-task contamination.

### The LLM needs to produce meaningfully different outputs between passes (Logic unstated assumption #1)
This is listed as an unstated assumption, but it is not unstated -- it is the explicit premise of the sensorium document. The two-pass architecture is designed around the empirical observation that LLMs condition strongly on their input context. An LLM asked "predict what the human will do" with only context and memories will produce a different distribution of outputs than the same LLM given context, memories, and a specific artifact. This is not a controversial claim about LLM capabilities -- it is how conditional generation works. The concern that the prior might be dominated by instruction-following ("always output cautious escalate") or by retrieved memories is a prompt engineering problem, not an architectural one. If the prior is always "escalate," that is visible in the data and indicates the prompt needs tuning, not that the two-pass architecture is flawed.

### Cosine similarity is not spreading activation (Logic non sequitur #4, Logic unstated assumption #2)
The critic is right that the mechanism is different from ACT-R's fan effect, and the "equivalent of low-fan spreading activation" phrasing overstates the mapping. But the documents are clear that embeddings replace spreading activation -- they don't claim to replicate it exactly. The mapping document (line 124) explicitly says: "TeaParty design -- not an ACT-R equation; replaces ACT-R's spreading activation with embedding similarity." The claim is that embeddings serve the same role (context sensitivity) via a different mechanism, not that they reproduce the fan effect's mathematical properties. The "low-fan" analogy should be softened to avoid implying mechanical equivalence, but the substitution itself is defensible: embeddings provide context-sensitive retrieval without requiring a pre-built associative graph, which is an advantage for a system where associations are learned from unstructured text.

### EMA and ACT-R memory may not be cleanly separable (Logic unstated assumption #5)
The critic raises a legitimate point: if EMA detects that upstream quality has degraded, a rational proxy should adjust its scrutiny. The document's clean separation (EMA monitors, ACT-R decides) is too rigid. But the information does flow, just indirectly. If upstream quality degrades, the proxy will encounter more corrections and more surprises, which produce more correction chunks with richer salience deltas. The memory system naturally becomes more skeptical as corrections accumulate. The EMA trend confirms what the memory is already reflecting. The separation is about mechanism (don't use a scalar to gate decisions), not about information (the proxy should ignore environmental trends). That said, the document could explicitly note that EMA trends might trigger operational actions (alerting the team, adjusting dispatch configuration) even though they don't directly alter the proxy's per-gate decision process.

### Multi-dimensional embedding average vs. single blended embedding (HM #2)
The critic asks for evidence that 5 separate embeddings retrieve better than 1 blended embedding. This is a fair challenge, and the document does not provide that evidence. But the architectural argument is not about retrieval quality of individual queries -- it is about retrieval flexibility. A single blended embedding of "PLAN_ASSERT + security + missing rollback + human said add rollback" produces one vector that captures all of those concepts blended together. You can only query it with another blended vector. Five independent embeddings let you query by situation alone, by salience alone, or by intersection. The value is in the ability to ask different questions of the same memory, not in any single query being better. Whether this flexibility matters enough to justify 5x the embedding cost is an empirical question that should be tested in Phase 1. The document should commit to comparing multi-dimensional vs. single-embedding retrieval quality during shadow mode.

## Clarifications

### "Salient percepts" are empty for most chunks but this does not break learned attention (Logic #2)
The critic observes that if most gates produce no surprise, most chunks have empty salience fields, and the learned attention model is "built on a minority of data." This is not a bug -- it is the design working as intended. Learned attention is supposed to be built from surprising interactions, not from all interactions. A human who reviews 100 plans and is surprised by 15 of them has learned attention from those 15 surprising cases, not from the 85 routine approvals. The routine approvals reinforce the prior (making it more accurate), while the surprising cases build the attention model (what to look for). The sensorium document could be clearer that learned attention is intentionally built from the minority of interactions that produce surprise, and that this is a feature of attention as a concept -- you attend to what is unexpected, not to what is routine.

### "Corrections are more informative" (Logic non sequitur #1)
The critic says that having more text does not mean more precise retrieval. This is correct as a general statement about embeddings. But the document's argument is narrower: corrections include a delta (what was wrong), which is a specific, targeted piece of text describing a particular failure mode. An approval has no delta -- it says "this was fine." The embedding of "missing rollback strategy for database migration" is more specific than the embedding of "approved." The argument is not that longer text produces better embeddings -- it is that error-describing text is inherently more specific than success-confirming text, because errors have content and successes are absences. This is not a category error; it is an observation about the asymmetric information content of corrections vs. approvals. The phrasing could be tighter, though -- "richer associations" should be "more specific associations" to avoid the implication that quantity equals quality.

### "Activation-weighted embedding retrieval is deployed at scale" (Factcheck, unverifiable)
The factcheck correctly notes this claim is unverifiable for Claude's own memory. The document should not claim specific knowledge of Claude's internal architecture. The broader claim -- that combining activation-like signals (recency, frequency) with embedding similarity for retrieval is a well-established pattern in production AI systems -- is defensible and widely practiced (e.g., in recommendation systems, search ranking, conversational memory systems). The specific reference to Claude should be removed; the general claim can stay with broader citations.

### Cold start problem (HM #5)
The critic says the design hand-waves the cold start. The document does address this -- "I don't know this human, so I must ask them everything" -- but the critic is right that the mechanical behavior is unspecified. However, this is simpler than it appears. With zero chunks in memory, retrieval returns nothing. The proxy's LLM receives no memories and must reason from the system prompt and artifact alone. This is equivalent to the proxy escalating every gate to the human, which is the correct cold-start behavior: when you know nothing about the human, ask them. The Laplace prior in the current system serves the same function (start uncertain, converge toward confidence). The difference is that the new system's cold start produces dialog (the proxy asks questions because it has no memories to draw on), while the current system's cold start produces escalation (the proxy punts because the scalar is below threshold). Both are conservative; the new system's conservatism is more informative because it generates interaction data that builds the memory. The document should state this explicitly rather than leaving it implicit.

### The activation threshold is self-regulating (Logic unstated assumption #4)
The critic questions whether tau = -0.5 will naturally admit 20-50 chunks as memory grows. The specific number is less important than the mechanism: as more chunks compete for activation, each chunk's reinforcement is spread thinner (the interaction counter advances but traces are added only to loaded chunks and explicitly reinforced ones), so old unreinforced chunks fall below threshold. The 20-50 figure is an estimate, not a guarantee. The document should acknowledge that the actual loading count depends on interaction tempo and reinforcement patterns, and that tau may need adjustment based on observed distributions in shadow mode. But the self-regulation claim is directionally correct: a fixed threshold on a decaying activation function will naturally limit how many chunks stay above it as the total population grows.

### Anderson & Schooler's event-based analysis (Logic non sequitur #3)
The critic says the proxy's interaction counter is not the same as Anderson & Schooler's environmental event stream. This is true -- the proxy's events are a sparse, self-selected subset of the world's events. But the document's claim is more modest than the critic reads it: the claim is that interaction-based time (rather than wall-clock time) is the right unit because the empirical basis for the decay parameters was event-based. This does not require that proxy interactions have identical statistical structure to newspaper headlines. It requires only that the power-law form and the event-based framing are appropriate -- and they are, because the alternative (wall-clock time) would produce decay during idle periods when no information arrived, which is clearly wrong. The specific exponent (d = 0.5) is a separate question from the choice of event-based time, and the document should separate these two claims.
